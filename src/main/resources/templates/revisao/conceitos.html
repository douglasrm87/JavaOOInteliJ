🎯 8 Pilares da Orientação a Objetos

1️⃣ Encapsulamento Proteger dados internos

🔒 1. Encapsulamento

Encapsulamento é o princípio de ocultar detalhes internos de uma classe, permitindo acesso apenas através de métodos públicos. Protege dados e garante integridade.

// ❌ Sem encapsulamento
class Pessoa {
  public String nome;
  public int idade; // Qualquer um pode mudar para -50!
}

// ✅ Com encapsulamento
class Pessoa {
  private String nome;
  private int idade; // Protegido
  
  public void setIdade(int idade) {
    if (idade > 0 && idade < 150) {
      this.idade = idade; // Valida antes de atribuir
    }
  }
}

💡 Exemplo no projeto: Classe Carrinho encapsula lista private List<Produto>

👨‍👧‍👦 2. Herança

Herança permite que uma classe herde características de outra, reutilizando código e criando hierarquias. Usa-se quando há relação "é um/uma".

// Superclasse abstrata
abstract class Produto {
  protected String nome;
  protected double preco;
  
  abstract void exibirDetalhes();
}

// Subclasse concreta herda de Produto
class Camisa extends Produto {
  private String tamanho;
  
  @Override
  void exibirDetalhes() {
    System.out.println(nome + " - Tam: " + tamanho);
  }
}

💡 Vantagem: Camisa herda nome, preco, getPreco() de Produto

🎭 3. Polimorfismo

Polimorfismo ("muitas formas") permite que objetos diferentes respondam ao mesmo comando de formas diferentes. Código genérico funciona com múltiplos tipos.

// Interface define contrato
interface MetodoPagamento {
  void processar(double valor) throws PagamentoException;
}

// Diferentes implementações
class CartaoCredito implements MetodoPagamento {
  public void processar(double valor) { /* valida cartão */ }
}

class PIX implements MetodoPagamento {
  public void processar(double valor) { /* valida PIX */ }
}

// Código polimórfico - funciona com qualquer MetodoPagamento
void fecharPedido(MetodoPagamento metodo) {
  metodo.processar(100.00); // Qual será executado?
}

💡 Magic: Chamada metodo.processar() executa versão correta em tempo de execução!

🔍 4. Abstração

Abstração é mostrar apenas o essencial, escondendo complexidade. Interfaces e classes abstratas definem "contrato" sem detalhar implementação.

// Interface abstrata - define O QUÊ
public interface MetodoPagamento {
  void processar(double valor) throws PagamentoException;
}

// Classe abstrata - define O QUÊ + estrutura
public abstract class Produto {
  protected String nome;
  abstract void exibirDetalhes(); // Subclasses implementam COMO
}

💡 Benefício: Quem usa não precisa saber COMO funciona, apenas O QUÊ faz

🧩 5. Composição

Composição cria objetos contendo outros objetos (relação "tem um"). Mais flexível que herança em muitos casos.

// Composição: Pedido TEM UM Carrinho
class Pedido {
  private Carrinho carrinho; // Campo de composição
  private StatusPedido status;
  private double valorTotal;
  
  public Pedido() {
    this.carrinho = new Carrinho(); // Cria objeto interno
  }
}

💡 Diferença de Herança: Herança é "é-um", Composição é "tem-um"

📦 6. Agregação

Agregação é tipo composição, mas objetos agregados podem existir independentemente. É uma relação mais fraca ("referencia").

// Agregação: Carrinho REFERENCIA Produtos
class Carrinho {
  private List<Produto> itens; // Agregação
  
  public void adicionarItem(Produto p) {
    itens.add(p); // Produto pode existir sem Carrinho
  }
}

💡 Diferença: Se Carrinho for deletado, Produtos continuam existindo!

📚 7. Collections API (Coleções)

Collections são estruturas que armazenam múltiplos objetos. List, Set, Map são interfaces que definem comportamentos diferentes.

// List - ordem importa
List<Produto> itens = new ArrayList<>();
itens.add(new Camisa(...)); // Index 0
itens.add(new Camisa(...)); // Index 1

// Set - sem duplicatas
Set<String> tags = new HashSet<>();
tags.add("promocao"); // OK
tags.add("promocao"); // Ignorado (já existe)

// Map - chave/valor
Map<String, Produto> estoque = new HashMap<>();
estoque.put("CAMISA_001", new Camisa(...));

💡 Projeto: Carrinho usa List<Produto> para manter ordem

🌊 8. Streams API (Programação Funcional)

Streams permite processar coleções com programação funcional. Operações como map, filter, reduce transformam dados de forma elegante.

// Sem Streams
double total = 0;
for (Produto p : itens) {
  total += p.getPreco();
}

// Com Streams ✨
double total = itens.stream()
  .mapToDouble(Produto::getPreco)
  .sum();

// Mais complexo: filtrar + mapear + somar
double caros = itens.stream()
  .filter(p -> p.getPreco() > 50)
  .mapToDouble(Produto::getPreco)
  .sum();

💡 Vantagem: Mais legível, conciso, e permite otimizações automáticas

🎨 Padrões de Design Demonstrados

Strategy Pattern

Diferentes algoritmos de pagamento (CartaoCredito, PIX, PagamentoPix) implementam interface MetodoPagamento

Template Method

Classe abstrata Produto define estrutura comum, subclasses (Camisa) implementam detalhes com @Override exibirDetalhes()

State Pattern

Enum StatusPedido com valores (AGUARDANDO_PAGAMENTO, PAGO, ENTREGUE) representa estados do pedido

MVC Pattern

Separação clara: Controllers roteiam, Services processam, Views (Thymeleaf) renderizam
